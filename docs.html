Detailed Model Maker

Detailed Model Maker is a Roblox Studio plugin for prompt-driven 3D model generation with built-in previewing, collision workflows, storage handoff, UI theming, and an in-plugin guidebook. Its job is to give creators a single panel where they can define a request, inspect the result, adjust runtime-readiness, and move generated assets into the correct play-mode storage flow.

Primary Use

Generate and iterate on detailed Roblox models from text prompts inside Studio.

UI theme system, preview viewport, storage pipeline, guidebook, collision tuning, and runtime preparation.

Overview

This plugin is built around a simple but practical premise: text prompt in, reviewable Roblox model out. Instead of treating generation as a one-shot action, the plugin treats it as a pipeline. You define the request, preview it, inspect collision assumptions, and then deliberately decide what should be stored for play-mode regeneration.

Prompt Driven

Text-to-Model Workflow

Prompts, seeds, style bias, and schema settings shape the generation request directly from Studio.

Preview First

Validation Before Commit

Generated previews can be inspected in a dedicated viewport with view, lighting, and collision display controls.

Studio Native

Integrated Tooling

Settings, guidebook, preview widget, theme selector, and storage actions all live as Studio dock widgets.

Runtime Aware

Storage Handoff

The plugin distinguishes between editable workspace models and stored runtime-ready assets in ServerStorage.

Objectives

Purpose

What The Plugin Exists To Do

The plugin exists to reduce the friction between idea, generated geometry, inspection, and final storage. It is designed for creators who want to work quickly from prompts but still keep control over technical details like collider mode, anchoring, scale, preview quality, and runtime storage behavior.

Objective

What “Success” Looks Like

A successful run is not just “a model appeared.” A successful run means the generated model was previewed, checked, optionally corrected, and stored in a way that makes sense for live play. The plugin is structured to support that full loop.

Recommended Workflow

The plugin is easiest to understand as a five-step pipeline.

Step 1

Choose A Detail Level

Use Medium, High, or Ultra to establish an initial balance between fidelity, complexity, and collider behavior.

Step 2

Write The Request

Fill out prompt, seed, schema, collider mode, size, triangle budget, texture preference, base preference, and anchoring.

Step 3

Preview It

Open the preview widget, orbit the model, inspect stats, change display presets, and verify scale/collision readability.

Step 4

Generate The Final

Run the full generation once the preview direction is acceptable.

Step 5

Store Intentionally

Store only the generated models you actually want prepared for runtime regeneration or later editing.

Storage And Runtime Pipeline

One of the most important design choices in this plugin is that it separates editable and stored model states.

Editable State

Workspace Editing

When stored assets are pulled back out for editing, they are placed in Workspace/DetailedModelEditableModels. This is the plugin’s “safe editing” area for models you want to inspect or correct before storing again.

Stored State

ServerStorage Backups

Models prepared for play are moved through ServerStorage/DetailedModelPreviewBackups and the plugin writes runtime regeneration metadata so a fresh model can be produced during Play.

Important Collision And Texture Note

Important

Manual Collision Pass Required Before Storage

Because the mesh is generated first and the texturing/storage path completes later when the stored asset is loaded into a live server, the collision setup may need a manual correction before you store the model. If you want the final live-server result to use the intended collision, open the generated model after generation and set CollisionFidelity to Precise on the generated mesh parts before storing the model back into storage.

  • Generate the model.
  • Inspect the generated mesh parts in Studio.
  • Set CollisionFidelity = Precise where needed.
  • Store the model only after that manual collision correction is done.
Why This Matters

Visual Correctness Is Not The Same As Runtime Correctness

A generated model can look visually correct in Studio while still using the wrong gameplay collision shape in the live server. This note exists to prevent that mismatch.
